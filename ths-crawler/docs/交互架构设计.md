# ths-crawler 交互架构设计方案

> 版本：v1.0 | 创建：2026-07-02 | 基于豆包交互总纲 + 现有代码审计

---

## 一、项目定位论证

### 1.1 我们是什么

**ths-crawler 是一个响应式 Web SPA（Single Page Application），不是原生 APP。**

| 维度 | 当前状态 | 说明 |
|------|---------|------|
| 技术栈 | React 18 + Vite + TypeScript + Ant Design 5 | 纯前端 Web 工程，无原生壳 |
| 部署形态 | 静态资源（`dist/`）→ Nginx → Cloudflare Tunnel → `ths.thsflow.xyz` | 浏览器访问，无应用商店分发 |
| 移动端能力 | CSS 媒体查询 + `Grid.useBreakpoint()` | 768px 断点切换布局，**非原生组件** |
| 离线能力 | 无 Service Worker、无 manifest.json | 断网不可用 |
| 推送能力 | 无 | 不支持系统级推送 |
| 桌面入口 | 无 | 浏览器书签，非 PWA |

**一句话：用户在任何设备上打开 `https://ths.thsflow.xyz`，看到的是同一个 Web 应用，只是 PC 和手机上的布局不同。**

### 1.2 我们不是什么

- ❌ 不是同花顺 / 东方财富那样的原生 APP
- ❌ 不是微信小程序（无 `app.json`、无微信 SDK）
- ❌ 不是 PWA（无离线缓存、无桌面图标、无推送通知）
- ❌ 不是 React Native / Flutter / uni-app 跨端项目

### 1.3 为什么选择响应式 Web 而不是原生 APP

| 决策因素 | 响应式 Web（当前选择） | 原生 APP | 结论 |
|---------|---------------------|---------|------|
| 开发成本 | 一套代码，PC+手机共用 | 需 iOS + Android 双端 | Web 成本远低于原生 |
| 迭代速度 | `vite build` → `scp dist/` 即可上线 | 需应用商店审核（1-3天） | Web 分钟级上线 |
| 用户触达 | 分享链接即可访问 | 需下载安装 | Web 零门槛 |
| 技术栈匹配 | Java 后端开发者可维护 React | 需 Swift / Kotlin | React 学习成本远低于双端原生 |
| 当前阶段 | 个人工具/分析平台，非商业产品 | 商业产品才需要 | 过度投入不划算 |
| 豆包总纲交互 | 底部Tab、横向标签、汉堡抽屉均可在 Web 实现 | — | Web 完全能模拟 APP 交互 |

**结论：响应式 Web 是当前阶段的最优解，未来有商业化需求时再考虑 PWA → 原生 APP 的演进路径。**

### 2.2 当前双布局机制

```typescript
// App.tsx — 已有移动/PC响应式拆分
function AppLayout() {
  const screens = Grid.useBreakpoint();
  const isMobile = !screens.md; // md = 768px

  return isMobile ? <MobileLayout /> : <DesktopLayout />;
}
```

| 布局 | 导航形式 | 触发条件 |
|------|---------|---------|
| DesktopLayout | 顶部固定 Header + 横向菜单 | `window.innerWidth ≥ 768px` |
| MobileLayout | 底部固定 5 个 Tab（排行/趋势/共振/历史对比/日历） | `window.innerWidth < 768px` |

### 2.3 已有移动端适配能力

- ✅ 触摸优化：按钮最小 40px、Select 最小 40px
- ✅ 卡片化布局：`.mobile-flow-card` / `.mobile-resonance-card` 等专用样式
- ✅ 表格横向滚动：`overflow-x: auto` + `-webkit-overflow-scrolling: touch`
- ✅ 底部安全区：`padding-bottom: env(safe-area-inset-bottom)`
- ✅ 深色主题：全局 `#0d1520` 底色，Ant Design `darkAlgorithm`
- ✅ ECharts 深色适配

### 2.4 与豆包总纲的现有对齐

| 豆包总纲概念 | 现有实现 | 匹配度 |
|------------|---------|--------|
| 底部 Tab 导航 | `MobileLayout` 底部 5 个 Icon+Label | ✅ 80% — Tab 直接对应页面，非业务域 |

| 汉堡抽屉 | 无 | ❌ 需新建 `Drawer` 组件 |
| 深色主题 | 已完整覆盖 | ✅ 100% |

## 三、目标架构：豆包总纲 Web 化落地

### 3.1 总纲核心设计 → Web 实现映射

豆包总纲描述的是一套**原生金融 APP 三级导航体系**。我们在 Web 中的实现对等映射如下：

| 豆包总纲 | 原生实现 | Web 实现 | 组件 |
|---------|---------|---------|------|
| 一级导航（底部Tab） | `UITabBarController`（iOS）/ `BottomNavigationView`（Android） | 4 个固定底部按钮 + React Router 路由切换 | Ant Design 无对应，手写 `footer` |
| 二级导航（横向标签） | `UIPageViewController` + 手势（iOS）/ `ViewPager2`（Android） | Ant Design `Tabs` + `swipeable` | `<Tabs>` |
| 三级导航（汉堡抽屉） | `UINavigationController` 左侧抽屉（iOS）/ `NavigationDrawer`（Android） | Ant Design `Drawer` | `<Drawer>` |
| 手势滑动 | 原生手势识别 | CSS `touch-action` + Tabs `swipeable` | 浏览器原生支持 |
| 页面预加载 | `viewDidLoad` 提前 | `React.lazy` + `Suspense` | — |

**关键结论：豆包总纲描述的所有交互模式，在 Web 中都有成熟的组件级对等实现，不需要原生能力。**

## 四、技术决策论证

### 4.1 为什么不拆成两个独立项目

| 拆分方案 | 优点 | 缺点 | 判断 |
|---------|------|------|------|
| 独立 PC + Mobile 两个仓库 | 各自独立迭代，不受对方约束 | API 层双写、组件双写、深色主题双维护、Bug 修两遍 | ❌ 不选 |
| 单仓库，两套路由+布局 | 零冗余，5 个页面组件共享，API 层共享 | App.tsx 稍复杂（可管理） | ✅ 当前方案 |

**核心判断：5 个核心页面组件（排行/趋势/共振/历史对比/日历）在 PC 和 Mobile 上功能完全一致，仅容器不同。拆分会引入不必要的维护负担。**

### 4.2 为什么不做 React Native / uni-app

| 方案 | 适用场景 | 本项目匹配度 |
|------|---------|------------|
| React Native | 需要调用原生能力（相机、推送、蓝牙） | ❌ 本项目纯数据展示，无原生依赖 |
| uni-app / Taro | 需要同时出微信小程序 + H5 + APP | ❌ 当前无小程序需求，引入框架增加复杂度 |
| Flutter | 高性能动画 + 跨端统一 UI | ❌ 学习成本高，与 Java 技术栈不匹配 |
| **响应式 React Web（当前）** | 数据展示类产品，一套代码多端 | ✅ 完美匹配 |

### 4.3 PWA 是什么时候该加

PWA 是在现有 Web 项目上加两个文件（`manifest.json` + Service Worker），不需要改任何业务代码。建议 **Phase 2 加 PWA**，收益明确：

| PWA 能力 | 作用 | 当前是否必要 |
|---------|------|------------|
| 添加到桌面 | 用户可从手机桌面图标打开，体验接近原生 APP | V1.0 非必须，V1.1 加 |
| 离线缓存 | 无网络时展示上一次缓存的数据 | 金融数据强时效，离线意义有限 |
| 推送通知 | 异动提醒、资金异动推送 | 依赖后端，P2 |

## 五、分阶段演进路线

```
现在 ────────→ Phase 1 ────────→ Phase 2 ────────→ Phase 3
响应式Web      导航重构         PWA化             决策平台
(已有)         (豆包总纲落地)    (桌面入口+离线)     (工具矩阵)
```

### Phase 1：导航架构重构（V1.0，1-2天）

**目标：** 豆包总纲 Web 化落地，手机浏览器体验接近原生 APP 交互。

**变更：**
- App.tsx 重写：4 一级 Tab + 资金分析内二级 Tabs + 汉堡 Drawer
- 新建 `FundFlowLayout.tsx`：二级标签容器，承接 5 个现有页面
- 新建 `HamburgerDrawer.tsx`：左侧抽屉（账号/设置/帮助）
- 新建 3 个占位页：`Home.tsx` / `Tools.tsx` / `Mine.tsx`
- 旧路由 `/trend` 等保留重定向，不破坏书签

**不变：**
- 5 个页面组件（FlowRanking / TrendAnalysis / ResonanceSignal / HistoryAnalysis / TradeCalendar）零改动
- `api/index.ts` 零改动
- `vite.config.ts` 零改动
- 深色主题 CSS 零改动

### Phase 2：PWA + 首页数据（V1.1，1周）

**目标：** 手机桌面图标 + 市场概览数据。

**变更：**
- 添加 `manifest.json`：APP 名称、图标、全屏模式
- 添加 Service Worker：静态资源缓存
- 后端新增 `GET /api/market/overview`：大盘指数 + 涨跌概览
- `Home.tsx` 接入真实数据

### Phase 3：决策工具矩阵（V2.0，P2）

**目标：** `Tools` Tab 上线，收纳选股/估值/回测等工具。

**变更：**
- 新建工具页面（宫格入口 + 分类标签）
- 后端新增对应 API
- 预留自定义 Tab 排序能力

## 七、架构约束与边界

1. **不引入移动端专用框架**（React Native / Taro / uni-app）——保持纯 Web 技术栈
2. **不拆分仓库**——PC 和 Mobile 共享 `ths-crawler-ui`，仅通过布局组件区分
3. **API 层不做任何改动**——现有 `api/index.ts` 的 9 个接口是前后端契约，导航重构不触及
4. **现有 CSS 不做减法**——`.mobile-flow-card` 等移动端样式全量保留，新导航叠加即可
5. **不引入新的 npm 依赖**——Ant Design 5 自带 Tabs、Drawer、Layout，无需额外安装
6. **GitHub 分支策略**：在 `feat/nav-restructure` 分支开发，验证通过后合入 `dev`
