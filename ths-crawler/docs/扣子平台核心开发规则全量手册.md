## 核心开发铁律集合
1. 沙箱写代码，云电脑只运行：所有文件编写修改在沙箱完成，云电脑仅做编译、部署、启动、验证、git pull，不做任何文件编辑
2. 编译通过后先push GitHub，再上传项目空间：GitHub为唯一代码分发中枢，优先级高于项目空间，防止沙箱崩溃丢数据
3. GitHub推送范围判断：影响编译/部署/运行的文件必须push；临时分析/用户上传数据仅留项目空间
4. 沙箱崩溃/重启后代码基线只能从GitHub拉取，禁止从项目空间拼凑
5. 项目空间写入路径必须与GitHub仓库结构完全一致，禁止截断路径产生脏路径
6. 测试执行分工：简单单元测试沙箱跑mvn test；Spring Boot启动、集成测试直接push后云电脑拉取运行
7. 云电脑禁止写/改任何文件内容，包括脚本，无例外
8. 问题诊断优先静态代码分析，修复验证优先单元测试：静态定位bug→写失败测试复现→改代码过绿灯→编译push→云电脑验证
9. 非任务范围问题不顺手修，先报告用户再决定
10. worklog每日23:45自动归档，带类别标签+重要性等级，仅P0/P1两个等级
11. 非代码文件统一放`/ths-crawler/docs/`，禁止放`/docs/`：项目文档、分析报告、复盘、攻略等非代码产物，上传到项目空间时必须走`/ths-crawler/docs/`路径，与GitHub仓库结构保持一致
12. `/docs/`目录为历史遗留，禁止碰：禁止上传、更新、读取、删除`/docs/`下的任何文件。该目录仅作历史存档，所有新文档和更新一律走`/ths-crawler/docs/`

## 其他详情
- GitHub v2工作流全步骤、SSH over 443优化配置、Flyway接入细节、ths-crawler项目全技术链路详情等内容均收录在此文件，作为完整参考手册
13. 架构变更后必须同步所有文档中的旧引用：类名重命名、包结构变更、模块删除后，必须扫描所有文档（拆解书、需求清单、复盘报告、联调现状等）中的旧引用并同步更新，避免文档误导后续开发
14. 需求清单与状态文档职责分离：需求清单（需求清单.md）为需求基线，不勾选完成状态；实时进度见 P0任务vs前后端联调现状.md，两份文档各司其职
15. TDD测试执行优化：mvn test 只跑单元测试（*Test.java），mvn verify 跑集成测试（*IT.java，需要Playwright/真实网络）；绿过的测试不重复跑，只跑变更相关的测试，避免浪费时间
16. 封禁检测优先用HTTP状态码：不要对HTML全文做子串匹配来判断是否被封（容易误判），优先用HTTP状态码（403/429=封禁），200直接放行，数据落地优先
