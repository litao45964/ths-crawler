> 2026-06-25 春风 & 小伍 共同整理
> 触发场景：云电脑重启后部署全链路瘫痪，排查恢复耗时2小时+，本质是1分钟的工作

## 一、重启后会发生什么

| 组件 | 重启后状态 | 根因 | 恢复难度 |
|------|-----------|------|----------|
| Java 17 | ❌ 回退到Java 11 | apt安装的openjdk-17-jdk未持久化，或update-alternatives未生效 | 中（需重装+设默认） |
| /var/run/mysqld/ | ❌ 目录丢失 | /var/run是tmpfs，重启自动清空 | 低（mkdir+chown） |
| MySQL | ❌ 停止 | 依赖/var/run/mysqld/目录，目录丢失则启动失败 | 低（目录重建后启动） |
| Redis | ❌ 停止 | 无自启配置 | 低（一条命令） |
| Spring Boot | ❌ 停止 | 依赖Java 17+MySQL+Redis，上游全挂 | 低（上游恢复后启动） |
| Nginx | ❌ 停止 | 无自启配置 | 低（一条命令） |
| Cloudflared | ❌ 停止 | 无自启配置 | 低（一条命令） |
| git仓库代码 | ✅ 不丢失 | /root/ths-crawler-repo/在持久磁盘上 | — |
| MySQL数据 | ✅ 不丢失 | 数据文件在持久磁盘上 | — |
| crontab @reboot | ✅ 不丢失 | startup.sh已注册 | — |

**核心结论：重启后几乎全部服务瘫痪，但数据和代码不丢。**

## 二、一键恢复脚本（重启后直接跑）

```bash
#!/bin/bash
# 云电脑重启后一键恢复脚本
# 用法：bash /root/restart-after-reboot.sh
set -e

echo "===== 1. 恢复Java 17 ====="
# 检查Java版本，如果不是17则安装
if java -version 2>&1 | grep -q "11\."; then
  echo "Java 11 detected, installing Java 17..."
  apt-get install -y openjdk-17-jdk
  export JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64
  export PATH=$JAVA_HOME/bin:$PATH
fi
java -version 2>&1 | head -1

echo "===== 2. 恢复MySQL ====="
mkdir -p /var/run/mysqld
chown mysql:mysql /var/run/mysqld
if ! mysqladmin -uroot -proot ping 2>/dev/null; then
  mysqld --user=mysql --daemonize
  sleep 5
  mysqladmin -uroot -proot ping
fi

echo "===== 3. 恢复Redis ====="
if ! redis-cli ping 2>/dev/null | grep -q PONG; then
  redis-server --daemonize yes
  redis-cli ping
fi

echo "===== 4. 拉取最新代码 ====="
cd /root/ths-crawler-repo
git pull

echo "===== 5. 编译 ====="
cd /root/ths-crawler-repo/ths-crawler
mvn package -DskipTests

echo "===== 6. 启动Spring Boot ====="
# 先杀旧进程
pkill -f ths-crawler-1.0.0-SNAPSHOT.jar 2>/dev/null || true
sleep 2
nohup java -jar target/ths-crawler-1.0.0-SNAPSHOT.jar --spring.profiles.active=prod > /root/ths-crawler-app.log 2>&1 &
sleep 10

echo "===== 7. 启动Nginx ====="
if ! pgrep -x nginx > /dev/null; then
  nginx
fi

echo "===== 8. 启动Cloudflared ====="
if ! pgrep -f "cloudflared tunnel" > /dev/null; then
  nohup cloudflared tunnel run ths-flow > /root/cf.log 2>&1 &
fi

echo "===== 9. 验证 ====="
sleep 5
echo "--- API验证 ---"
curl -s http://localhost:8100/api/industry-flow/industries | head -100
echo ""
echo "--- 公网验证 ---"
curl -s https://ths.thsflow.xyz/api/industry-flow/industries | head -100
echo ""
echo "===== 全部恢复完成 ====="
```

## 三、最关键的一条：bash连接云电脑的正确写法

```
❌ bash(desktop_name="爬虫云服务器")   → 设备未找到
❌ bash(desktop_name="VM-43-194-ubuntu") → 设备未找到
✅ bash(desktop_name="cloud")           → 直接在云电脑执行
```

**这是今天最大的坑。** 因为设备名写错，导致：
- 尝试 computer_use 绕路 → 权限不够、sudo受限、管道符不支持
- 尝试 SSH 从 Mac 连 → 端口22/443都超时
- 尝试上传 jar 到项目空间 → 502网关错误
- 反复重试累计耗时 2 小时+

**而正确答案 `bash(desktop_name="cloud")` 就写在 dev-workflow-guide-v2.md 里。**

### 教训：遇到设备找不到，第一反应应该是查已有文档的精确写法，而不是猜测和尝试各种变体。

## 四、重启后的恢复决策树

```
云电脑重启了
  │
  ├─ Q1: 能用bash(desktop_name="cloud")连上吗？
  │   ├─ YES → 走一键恢复脚本，5分钟搞定
  │   └─ NO  → 只能用computer_use（慢、受限）或Coze控制台终端
  │
  ├─ Q2: Java版本对吗？
  │   ├─ Java 17 → 继续
  │   └─ Java 11 → apt-get install -y openjdk-17-jdk
  │                 export JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64
  │                 export PATH=$JAVA_HOME/bin:$PATH
  │
  ├─ Q3: MySQL能启动吗？
  │   ├─ YES → 继续
  │   └─ NO  → mkdir -p /var/run/mysqld && chown mysql:mysql /var/run/mysqld
  │            mysqld --user=mysql --daemonize
  │
  ├─ Q4: 需要拉新代码吗？
  │   ├─ YES → cd /root/ths-crawler-repo && git pull
  │   └─ NO  → 跳过
  │
  └─ Q5: 需要重新编译吗？
      ├─ YES → cd /root/ths-crawler-repo/ths-crawler && mvn package -DskipTests
      └─ NO  → 直接重启服务
```

## 五、computer_use vs bash 对比（云电脑场景）

| 维度 | bash(desktop_name="cloud") | computer_use |
|------|---------------------------|--------------|
| 执行速度 | 秒级返回 | 10-30秒/条命令 |
| sudo权限 | root用户，大部分不需要sudo | sudo只允许apt/dpkg |
| 管道符/重定向 | ✅ 支持 | ❌ 不支持（会被拦截） |
| git pull | ✅ 可执行 | ❌ .git权限拒绝 |
| mvn编译 | ✅ 可执行 | ⚠️ 可执行但连续失败触发hard_fail_guard |
| 交互式操作 | ❌ 不适合 | ✅ 浏览器/GUI操作 |
| 文件系统 | 与SSH登录一致 | 容器化视图，可能与SSH不同 |
| 启动开销 | 几乎无 | 每次启动需初始化会话 |

**原则：能用bash就用bash，只有浏览器/GUI操作才用computer_use。**
